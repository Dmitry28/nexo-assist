# Product Plan — Listing Watch Bot

Дорожная карта: превращаем прототип `land-scraper` в многопользовательский
сервис на скелете **nexo-assist** (NestJS 11). Логику парсинга портируем из
land-scraper с рефакторингом. Развиваем по фазам — от простого локального к
распределённому. Как продукт работает — в [PRODUCT.md](PRODUCT.md).

---

## 1. Принципы

- По фазам: от простого локального к сложному распределённому.
- Best practices: Docker/k8s, секреты, observability (otel/prometheus/pino в
  скелете), миграции БД, тесты на каждый адаптер.
- Каждая фаза: на входе детально изучаем и обсуждаем варианты; этот документ —
  только высокоуровневый каркас.

## 2. Решения

| Тема                    | Решение                                                                                 |
| ----------------------- | --------------------------------------------------------------------------------------- |
| Telegram UX (старт)     | Личный чат, управление **кнопками**, не `/add`.                                         |
| Telegram UX (позже)     | Группа + forum-топики (топик на подписку). Возможно Mini App.                           |
| События уведомлений     | Сейчас только **new**; removed/price — Фаза 2.4; выбор на подписку — Фаза 7.            |
| Хостинг                 | Бесплатно/дёшево + практика k8s. Oracle Free + k3s (варианты уточнить на входе Фазы 5). |
| Трекер новых источников | GitHub Issues → LLM генерит адаптер.                                                    |
| Репозиторий             | `nexo-assist` (этот репо и есть продукт).                                               |
| Bot transport           | Переключаемый: long-polling (Фазы 1–4) → webhook на k8s (Фаза 5).                       |
| Postgres                | Локально — Docker; staging/prod — Neon; опц. in-cluster (CloudNativePG) на k8s.         |
| ORM                     | TypeORM. Миграции только генерируемые, `synchronize: true` в проде запрещён.            |
| Git-флоу                | От `dev` → мерж в `dev` → тест → только потом `main`.                                   |

- Ежедневный крон (`WATCH_CRON`); dev-команда `/check`.
- Перенесено в Фазу 2: realt-адаптер, инкрементальная пагинация (стоп на виденных).

### Фаза 3 — БД и мультипользовательность

Цель: заменить in-memory на Postgres и вынести юзеров в отдельную сущность.
Минимально: без `listings`/price-history и без дедупа-по-URL — это отдельный
шаг позже. Делаем под-шагами.

**Решения фазы:**

- Один `DATABASE_URL` (Neon отдаёт URL). Локально — Docker; staging/prod — Neon
  (SSL обязателен, использовать pooled-endpoint; учесть дроп idle-коннектов).
- Только генерируемые миграции; `synchronize: false` всегда. Отдельный
  `data-source.ts` для CLI (не переиспользуем `TypeOrmModule.forRootAsync`).
- `source` — `varchar`, не PG-enum (новый источник не должен требовать миграцию типа).
- UUID PK для `users`/`subscriptions` (уже так); `seen_listings` — составной PK
  `(subscription_id, external_id)` без суррогатного id (он же и индекс проверки).
  Timestamps; `baselinedAt` nullable. FK — `ON DELETE CASCADE`.
- Тесты: юнит — лёгкий in-memory fake репозитория (быстро, без БД); e2e/интеграция —
  реальный Postgres из docker-compose (сервис в CI).

**3.1 — Инфра БД — сделано.** 3.1a: Postgres в `docker-compose.yml` (+ healthcheck)
и скрипты `db:up/down/reset`. 3.1b: зависимости (`@nestjs/typeorm typeorm pg`),
`DATABASE_URL` в env.validation + configuration + `.env.example`,
`TypeOrmModule.forRootAsync` (inject `AppConfig`, `autoLoadEntities`,
`synchronize:false`, SSL для не-локальных), readiness-проба `db.pingCheck`, сервис
Postgres в CI. (k8s Secret для `DATABASE_URL` — в Фазе 5, деплой.)

**3.2 — Схема + первая миграция — сделано.** Сущности `Subscription` (uuid PK,
`telegramUserId`, source, url, createdAt, `baselinedAt`) и `SeenListing` — составной
PK `(subscriptionId, externalId)` (он же индекс проверки «видели/нет»), `seenAt`, FK
→ subscription `ON DELETE CASCADE`. `data-source.ts` (CLI) + скрипты
`migration:generate|run|revert|show` (ts-node/tsconfig-paths) + первая миграция
`InitSchema`. `User` + FK `subscription → user` + `forFeature` — в 3.3 (вместе с их
первым потребителем).

**3.3 — Перенос сервисов на БД (async) — сделано.** `SubscriptionsService` через
`@InjectRepository` + `TypeOrmModule.forFeature`; методы стали `async` → await в
`WatchService`, `telegram.handlers`, `watch.scheduler`. `getSeen` — запрос по кандидатам
прогона (`external_id = ANY(ids)`), не загрузка всего сета. `baseline` (seed +
mark-baselined) — в одной транзакции (`seedBaseline`). Удаление подписки каскадит
`seen` (FK). Тесты: слой данных → интеграционный на реальном Postgres (`test/`), юниты
`handlers`/`scheduler` — на моках коллабораторов. Ключ пока — `telegramUserId`.

**3.3b — Таблица `User` + профиль — сделано.** `User`-сущность (аккаунт: `telegramId`
unique — первая identity; username, имя, язык; веб-авторизация позже расширяет тот же
ряд); `Subscription` → `user` (eager, FK `ON DELETE CASCADE`), `telegramUserId` убран.
Upsert профиля при подписке (`ctx.from` → `users`). Миграция `AddUsers`. Ключ снаружи
остаётся telegram-id (сервис резолвит в user). Профиль пока не читается — собираем
заранее (при подписке `ctx.from` есть, потом не восстановить).

**3.4 — Чистка `seen` по окну — сделано.** Бот смотрит только первые несколько страниц
поиска (page-cap ≈ 150 самых свежих); всё, что уехало дальше, он больше не запрашивает —
значит и «новым» показаться не может. Поэтому храним только последние `N` виденных id на
подписку (`MAX_SEEN_PER_SUBSCRIPTION=300`, с запасом к page-cap), старые удаляем прунингом
после каждой вставки. Таблица на подписку ограничена навсегда, без риска повторно прислать
старое.

**3.5 — Лимиты / анти-абьюз — сделано.** Max подписок на юзера
(`MAX_SUBSCRIPTIONS_PER_USER=50`, проверка в `add`) + запрет дублей по точному URL;
дружелюбные сообщения в боте (лимит / «уже отслеживаешь»). Полная уникальность
`(user_id, normalized_url)` + DB-констрейнт — со шагом нормализации URL (Фаза 2.4).

**3.6 (опц.) — In-run fetch-кэш.** Несколько подписок на один URL в одном прогоне →
один fetch (Map на прогон). Дешёвый предшественник дедупа; можно перенести в Фазу 4.

**Отдельным шагом (позже, вне Фазы 3):** дедуп-по-URL (один скрейп на нормализованный
URL → fan-out) и вместе с ним `listings` + **снимки цен** (price history) — фундамент
для removed/price-событий (2.4), триггеров «цена упала» (Фаза 7) и аналитики (Фаза 8).

### Фаза 4 — Постоянный сервис + планировщик

- Always-on бот + отдельный scrape-worker по расписанию.
- **Поднять частоту проверок** (hourly → near-instant) после троттлинга/дедупа —
  скорость уведомления = главное конкурентное поле (у конкурентов real-time).
- Троттлинг/джиттер, бюджет прокси (scrapfly), backoff на бан.
- Telegram fan-out: очередь+троттлинг, дайджест вместо спама, 403 → пауза подписок.
- Изоляция сбоев: падение одного источника не рушит другие и не портит снимок;
  алерт админу при поломке адаптера.
- Остановка источников без подписчиков, чистка старых данных (free-tier).
- **Детект «мёртвой» подписки**: URL отдаёт 0/ошибку N прогонов подряд → уведомить
  юзера «обнови ссылку» + автопауза.
- Observability и health/readiness из скелета — подключить к воркеру;
  **продуктовые метрики** в Prometheus (юзеры/подписки/доставки/ошибки адаптеров);
  админ-команда `/stats` (юзеры/подписки/последний прогон).
- **OpenTelemetry + алерты по ошибкам**: экспорт трейсов/ошибок (OTLP) в бэкенд
  (Grafana Tempo/Sentry-подобный) + правила алертинга (spike ошибок скрейпа/доставки,
  рост latency, падение адаптера) с нотификацией (Telegram админ-чат / Alertmanager).
  OT уже инициализируется в `tracing.ts` — здесь добавляем экспорт и алерт-правила.

### Фаза 5 — Деплой в k8s (бесплатно)

- Dockerize (multi-stage), Helm/Kustomize, Secrets/ConfigMaps.
- **Миграции при деплое**: скомпилировать миграции в `dist` + JS data-source и гонять
  `migration:run` в рантайм-образе (initContainer / entrypoint) — сейчас миграции
  запускаются только с хоста/в dev-override. k8s Secret для `DATABASE_URL`; SSL с
  верификацией сертификата (Neon CA) — закрыть TODO в `app.module.ts`.
- Deployment (бот, webhook) + CronJob (скрейп) + Postgres (managed или CloudNativePG).
- Oracle Cloud Always Free + k3s. CI/CD: GitHub Actions → registry → deploy.
- Probes, ресурсные лимиты, graceful shutdown (в скелете есть).

### Фаза 6 — Неподдерживаемая ссылка → GitHub Issue → LLM-адаптер

- Распознавать и «голые» домены без `http://`; явное сообщение о неподдерживаемом
  источнике + кнопка «Запросить источник».
- Неизвестная ссылка → спросить «что детектить» → GitHub Issue по шаблону
  (url + описание + telegram_id заявителя).
- Ревью человеком → Claude генерит адаптер по контракту + фикстурам → уведомить заявителя.

### Фаза 7 — UX-апгрейды

- Доставка всего дайджеста батчами (несколько сообщений, общий кап ~100) вместо
  «…and N more»; кнопка «Показать ещё» для «Show current».
- Уточнить копи baseline: «сейчас N (это база, не шлю); новые буду присылать».
- Выбор event-типов на подписку.
- Частота на подписку: instant (~15–30 мин) / daily digest; **тихие часы** (не слать ночью).
- Пост-фильтры на подписку: min/max цена, ключевые слова, «только с фото»;
  ценовые триггеры («цена ниже X / упала на N%»).
- «Избранное»: следить за конкретным объявлением (цена/снятие); кнопки на карточке
  уведомления — ⭐ избранное / 🙈 скрыть.
- Индикатор здоровья в `/list`: «проверено сегодня 09:00, новых: 2» у подписки.
- Приватность: `/delete_me` (удалить все свои данные) + экспорт подписок.
- Валютное предпочтение BYN/USD на юзера.
- **Deep-links** (`t.me/bot?start=…`): кнопка «Поделиться поиском» (подписка в 1 тап),
  реферальный трекинг источников юзеров; связка с постами каналов.
- **Месячное подтверждение**: одно сообщение со списком подписок, «✅ Оставить все» /
  выборочно; нет ответа N дней → автопауза + возврат одной кнопкой (экономит скрейпинг).
  Позже — спрашивать только неактивных.
- **Цель доставки на подписку** (`deliveryTarget`): личка | свой канал | группа+топик
  (бот-админ сам заводит топик на подписку). Тот же механизм — для кураторских каналов.
- **Публичные кураторские каналы** (владелец): авто-постинг подборки по фикс-ссылке
  в публичный канал (напр. «Участки Гродно») → набор подписчиков → монетизация
  рекламой (Фаза 9). Параллельно с личными подписками.
- Telegram Mini App.
- i18n при необходимости.

### Технический бэклог (из ревью 2026-07)

Отложено осознанно — сделать при ближайшем касании соответствующего кода:

- **realt: страницы с `apolloState`.** Вживую часть поисковых URL рендерит
  `pageProps.apolloState` без `objects` — парсер видит их как пустые. Выяснить,
  какие URL так отвечают, и поддержать эту форму (или перейти на API realt).
- **`fetchHtml`: потоковый size-cap.** Chunked-ответ без `Content-Length` сейчас
  буферизуется целиком до проверки лимита; читать потоково с ранним abort. Там же:
  redirect-пиннинг сейчас проверяет финальный URL после перехода (blind-GET уже
  ушёл) — перейти на `redirect: 'manual'` с валидацией каждого хопа.
- **`WATCH_CRON`: настоящая cron-валидация** в env-схеме (сейчас только «5 полей»;
  мусор валит бут позже, в конструкторе CronJob).
- **Pending-подтверждения: TTL** (сейчас только cap 500, протухшие живут до
  вытеснения) — вместе с DB-слайсом.
- **Тестируемость бота:** инжектируемая фабрика `Bot`, чтобы покрыть enabled-путь
  `TelegramService` (start/notify); e2e на throttler (429).
- **CI:** порог coverage; `npm audit` без `continue-on-error`; smoke-тест
  собранного Docker-образа (`docker run` + health-check).
- **OTel:** дождаться flush в shutdown-хуке (сейчас `void sdk.shutdown()`).
